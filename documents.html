Analyse comparative — Bibles Techniques
Document de travail · 13 mars 2026 · Préparation sprint commun Nicolas × Vianney
✓ Ce qui a été réussi — commun aux deux

Les deux projets ont résolu le problème difficile : capturer de la connaissance technique multimodale (vidéo, photo, audio, PDF) et la rendre interrogeable en langage naturel. C'est le cœur du problème, et il est résolu dans les deux cas.

Bible Technique Nicolas — Torréfaction BIBAL Av12

Statut : En production — v1.0 stable depuis le 10/03/2026
Métriques : 48 fiches · 196 médias · 15+ sprints · 25+ bugs documentés

Architecture

Pipeline déterministe avec 2 modèles IA :

  • Gemini 2.5 Flash — perception multimodale (vidéo, photo, audio, texte)
  • Claude Sonnet 4 — raisonnement et structuration (2 passes anti-troncature)

Workflows n8n avec appels HTTP directs — paradigme "pipeline IA" plutôt qu'"agents autonomes".

Forces
  • ✓ Pipeline ingestion multimodale éprouvé (Gemini Files API + polling + retry HEVC)
  • ✓ Architecture 2 passes Claude avec staticData — anti-troncature
  • ✓ onError: continueErrorOutput sur tous les nœuds HTTP critiques
  • ✓ PROD/DEV séparés + scripts promotion + rollback
  • ✓ Documentation exhaustive : 7 fichiers MD + méthode de travail
  • ✓ Analytics de lacunes automatiques
  • ✓ Mémoire conversationnelle Telegram
Faiblesses
  • ⚠ RAG par mots-clés : limite réelle ~70 fiches avant dégradation
  • ⚠ Zéro chunking : 1 fiche = 1 bloc entier envoyé au LLM
  • ⚠ Aucune mesure de qualité des réponses
  • ⚠ Clés API en dur dans les nœuds
  • ⚠ Pas d'agents IA natifs (évolution planifiée)

Bible Technique Vianney — Distributeurs Bibal

Statut : Architecture posée — non déployée
Périmètre : Distributeurs café, snack, fontaine · 6 workflows n8n · 3 agents IA

Architecture multi-agents hiérarchiques

  • Agent Superviseur — orchestration, score de confiance 0–100, JSON strict
  • Agent Analyseur (agentTool) — 3 passes, 4 outils multimodaux
  • Agent Organisationnel (agentTool) — gestion Drive, arborescence 3 niveaux

Pattern agentTool / toolWorkflow / $fromAI() — les agents décident eux-mêmes des outils à appeler.

Forces
  • ✓ Architecture multi-agents propre et bien structurée
  • ✓ Score de confiance automatique 0–100 par fiche
  • ✓ BigQuery VECTOR_SEARCH — infrastructure déjà configurée
  • ✓ Organisation Drive auto-gérée par l'IA
  • ✓ Ingestion binaire directe (PDF, image, vidéo, audio)
Risques critiques
  • ✗ Aucun error handler — pertes silencieuses garanties
  • ✗ Pipeline embedding non connecté — RAG non fonctionnel
  • ✗ Credential placeholder REPLACE_WITH_CREDENTIAL_ID
  • ✗ Pas d'environnement DEV
  • ✗ Zéro documentation

Ce qui manque aux deux projets

#1 Chunking sémantique — levier de qualité n°1

1 fiche entière = 1 vecteur. La découpe par section (procédure / symptômes / causes) est la modification avec le plus fort impact : +30–50% de précision au retrieval, documenté dans la littérature RAG.

#2 Évaluation de la qualité — golden dataset

Aucun système ne mesure si ses réponses sont bonnes. Sans mesure, chaque sprint est une amélioration dans le vide. Minimum viable : 20–30 questions réelles avec réponses de référence.

#3 Hybrid Search — BM25 + vectoriel + RRF

Le vectoriel seul rate les termes exacts ("erreur E47", "Av12"). BM25 seul rate la sémantique ("machine qui claque" ≠ "bruit anormal moteur"). Il faut les deux, fusionnés par Reciprocal Rank Fusion.

#4 Feedback loop 👍/👎 + Détection d'incertitude

Un 👎 Telegram vaut plus qu'un sprint entier de tuning à l'aveugle. Et si les chunks récupérés ont un score faible, le LLM hallucine — un seuil de confiance (score < 0.72) doit bloquer la réponse.

Opportunité — Gemini Embedding 2 (sorti 10/03/2026)

⚡ Premier modèle d'embedding nativement multimodal

Texte, images, vidéos, audio, PDF dans un seul espace vectoriel. Plus besoin de transcrire la vidéo avant l'embedding — on embedde directement le média. Model ID : gemini-embedding-2-preview · Dims : 128–3072 (MRL).

⚠ Limites actuelles en Preview

Vidéo : 128s max · Audio : 80s max · PDF : 6 pages max · Incompatible avec text-embedding-004 (re-embedding requis). Ne pas utiliser aujourd'hui. Anticiper la migration avec une colonne embedding_model_version dans BigQuery.

⭐ Recommandation principale : Scénario B — Convergence

Ne pas repartir de zéro. Ne pas continuer en parallèle. Fusionner les deux projets : base opérationnelle de Nicolas + architecture agents de Vianney + BigQuery VECTOR_SEARCH (déjà configuré) + les fondations manquantes aux deux.

Trois scénarios évalués

A
Évolution incrémentale — Partir de Nicolas
4–6 semaines · Risque faible · Récompense modérée

Greffer BigQuery VECTOR_SEARCH sur la base Nicolas sans changer l'architecture.

  • ✓ Résout la saturation RAG (~70 fiches)
  • ✗ Pas d'agents IA, pas d'évaluation, pas de feedback loop

À choisir si : urgence uniquement sur la scalabilité, projets restent séparés.

B
Convergence ⭐ Recommandé
8–12 semaines · Risque modéré · Récompense élevée

Fusionner les deux systèmes + ajouter les fondations manquantes aux deux.

  • ✓ Capitalise sur 15+ sprints Nicolas sans les jeter
  • ✓ Intègre l'architecture avancée de Vianney
  • ✓ Pose les vraies fondations RAG (chunking, évaluation, hybrid search)
  • ✓ Prépare la migration Gemini Embedding 2
C
Greenfield microservice RAG
16–20 semaines · Risque élevé · Récompense maximale

n8n pour les déclencheurs uniquement, tout le RAG dans un service Python (FastAPI + LlamaIndex).

  • ✓ Testable, observable, évaluable avec RAGAS
  • ✗ Compétences Python requises, 4+ mois sans amélioration prod

À considérer dans 12–18 mois si volume > 500 fiches.

Architecture cible

Pipeline d'ingestion

Formulaire Airtable
  → Webhook n8n
    → Agent Superviseur (n8n LangChain)
        ├── Agent Analyseur (agentTool)
        │     → Gemini 2.5 Flash (multimodal)
        │       → Chunking sémantique (Code node)
        │         → Gemini embedding-001
        │           → BigQuery VECTOR_SEARCH (table chunks)
        ├── Agent Organisationnel (agentTool)
        │     → Google Drive (arborescence auto)
        └── Claude Sonnet 4
              → Fiche structurée Airtable + Score confiance 0–100

Pipeline de retrieval (bot Telegram)

Question opérateur (texte ou vocal)
  → Gemini Speech (si vocal)
    → Gemini embedding-001 (vecteur question)
      → BigQuery : VECTOR_SEARCH cosine + fulltext BM25
        → Fusion RRF (top 20)
          → Cohere Rerank v3 (top 5)
            → Seuil confiance (score < 0.72 → "Je ne sais pas")
              → Claude Sonnet 4 (génération réponse)
                → Réponse Telegram + bouton 👍/👎

  2. Métadonnées au chunk — filtrage pré-retrieval réduit le bruit
  3. Hybrid search BM25 + vectoriel — couvre termes exacts ET sémantique
  4. Évaluation systématique — golden dataset 20–30 questions
  5. Feedback loop 👍/👎 — signal terrain directement exploitable

Phase 1 — Fondations RAG v2

RAG-1 BigQuery — branchement pipeline embedding P1 · 1 jour
BigQuery VECTOR_SEARCH est déjà configuré. Le branchement n8n → Gemini embedding → BigQuery INSERT manque côté Nicolas.
  • Table chunks créée dans BigQuery (schéma commun aux deux projets)
  • Workflow n8n : à chaque nouvelle fiche → Gemini embedding-001 → INSERT chunks
  • Test : requête VECTOR_SEARCH retourne des résultats pertinents
RAG-2 Chunking sémantique des fiches P1 · 1 jour
1 fiche = 1 vecteur actuellement. La découpe par section multiplie la précision de retrieval.
  • Code node n8n : 1 fiche → N chunks (titre / procédure / symptômes / causes / matériel / notes)
  • Métadonnées au chunk : machine, section, score_confiance, embedding_model_version
  • Re-embedding des 48 fiches Nicolas existantes
RAG-3 Remplacement RAG mots-clés Nicolas P1 · 1 jour
Le bot Nicolas fait GET ALL fiches → filtre mots-clés → Claude. Saturation à ~70 fiches.
  • Nœud retrieval remplacé : question → embedding → BigQuery VECTOR_SEARCH → top 5 chunks → Claude
  • Test sur golden dataset (RAG-4)
  • Vérification : qualité ≥ ancienne méthode sur les 20 questions de référence
RAG-4 Golden dataset — évaluation initiale P1 · 2h humain
Sans mesure de base, on ne sait pas si on améliore ou on dégrade. Ce sprint est du travail humain — Nicolas et Vianney ensemble.
  • Fichier golden_dataset.json : 20–30 questions réelles (10 Nicolas + 10 Vianney + 10 communes)
  • Format : { question, reponse_reference, fiche_source, section_attendue }
  • Score de base mesuré : taux de retrieval correct + qualité réponse LLM-as-judge
RAG-5 Hybrid search BM25 + vectoriel + RRF P2 · 1 jour
  • Requête BigQuery combinant VECTOR_SEARCH cosine + fulltext search
  • Fusion RRF : score_final = 0.6 × score_vecteur + 0.4 × score_BM25
  • Benchmark sur golden dataset : hybrid vs vectoriel seul
RAG-6 Seuil de confiance + "je ne sais pas" P2 · 2h
  • Code node : si max_similarity_score < 0.72 → message de redirection + suggestion de créer un dossier
  • Test avec questions hors-sujet
RAG-7 Intégration agents Vianney dans pipeline Nicolas P2 · 2 jours
Nicolas utilise des appels HTTP directs. Migrer vers le pattern agentTool de Vianney.
  • Agent Superviseur dans workflow Nicolas
