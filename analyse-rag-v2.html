Section 1

Résumé exécutif

Cinq points à retenir avant de lire la suite.

✓ Ce qui a été réussi

Les deux projets ont résolu le problème difficile : capturer de la connaissance technique multimodale (vidéo, photo, audio, PDF) et la rendre interrogeable en langage naturel. C'est le cœur du problème, et il est résolu dans les deux cas. Le reste est de l'ingénierie.

⚠ Les deux projets sont des POCs qui atteignent leurs limites

Nicolas est en production avec 48 fiches — le RAG par mots-clés commence à saturer. Vianney a une architecture plus avancée mais aucun filet de sécurité ni pipeline d'alimentation opérationnel. Aucun des deux n'a de mesure de qualité des réponses.

→ La recommandation : Scénario B — Convergence

Partir de la base opérationnelle de Nicolas (DevOps, error handling, documentation), y intégrer l'architecture multi-agents de Vianney et un vrai moteur vectoriel (Supabase pgvector). Ni repartir de zéro, ni continuer en parallèle.

⚡ Opportunité externe : Gemini Embedding 2 (sorti le 10/03/2026)

Premier modèle d'embedding nativement multimodal — vidéo, photo, audio, texte dans le même espace vectoriel. Pas encore en production (Preview), mais l'architecture à poser aujourd'hui doit anticiper cette migration.

→ Le levier de qualité n°1 (souvent ignoré)

Le chunking sémantique — découper les fiches par section (procédure / symptôme / cause) plutôt qu'une fiche = un vecteur — est la modification qui améliorera le plus la qualité des réponses. Avant même le choix du LLM.

Section 4

Ce qui manque aux deux projets

Ces lacunes sont communes aux deux systèmes. Elles définissent l'agenda du RAG v2.

🔪 Chunking sémantique

Actuellement, une fiche complète = un seul vecteur ou un seul bloc envoyé au LLM. C'est la première cause de mauvaise réponse : si la question porte sur les symptômes, le système retrouve des fiches entières au lieu du paragraphe "symptômes" exact.

Impact estimé : +30–50% de précision au retrieval, documenté dans la littérature RAG.

// Structure cible par fiche
{
  chunk_id: "fiche_42_symptomes",
  fiche_id: "rec...",
  section: "symptomes",
  machine: "Av12",
  type: "Panne",
  contenu: "Bruit anormal au démarrage...",
  vecteur: [0.12, -0.04, ...]
}

📊 Évaluation de la qualité

Aucun système ne mesure si ses réponses sont bonnes. Sans évaluation, chaque sprint est une amélioration dans le vide — on ne sait pas si on régresse ou si on progresse.

Minimum viable : un "golden dataset" de 20–30 questions réelles avec réponses de référence, testé à chaque sprint (30 min de travail, impact permanent).

  • Faithfulness — la réponse est-elle fondée sur les fiches ?
  • Relevancy — les chunks récupérés sont-ils pertinents ?
  • Answer quality — LLM-as-judge sur les réponses

🔍 Hybrid Search

La recherche vectorielle seule rate les termes exacts : "erreur E47", "Av12", "roulement référence 6205". La recherche BM25 seule rate la sémantique : "machine qui claque" ≠ "bruit anormal moteur". Il faut les deux, fusionnés.

// Reciprocal Rank Fusion (RRF)
score_final =
  0.6 × score_vecteur_cosine
+ 0.4 × score_BM25_fulltext

👍 Feedback loop

Un 👎 Telegram est la donnée la plus précieuse du système. Actuellement ces signaux sont perdus. En les capturant, on peut identifier les fiches mal indexées, les chunks mal découpés, les questions non couvertes — sans aucun effort supplémentaire des opérateurs.

  • Signal explicite : bouton 👍/👎 après chaque réponse
  • Signal implicite : relance de la même question = insatisfaction
  • Sortie : dashboard lacunes + re-ranking par utilité

🛡 Détection d'incertitude

Si les chunks récupérés ont un score de similarité faible, le LLM va halluciner une réponse plausible au lieu de dire "je ne sais pas". Les deux systèmes n'ont aucun seuil de confiance sur le retrieval.

// Garde-fou à ajouter
if (max_similarity_score < 0.72) {
  return "Je n'ai pas de fiche sur ce sujet.
  Voulez-vous créer un dossier ?"
}

⚙️ Prompt versioning

Les prompts sont dans les nœuds n8n, non versionnés, non testables. Modifier un prompt sans version de référence rend toute régression impossible à identifier. Nicolas a partiellement résolu ça avec les fichiers MD de prompts, mais sans lien formel avec les nœuds.

  • Un fichier par prompt, versionné git
  • Numéro de version stocké dans le nœud n8n
  • Golden dataset rejoué après chaque modification de prompt

Section 6

Trois scénarios d'évolution

A
Évolution incrémentale — Partir de Nicolas
4–6 semaines · Risque faible · Récompense modérée

Greffer un moteur vectoriel sur la base existante de Nicolas sans changer l'architecture ni interrompre la production.

Ce qu'on fait

  • Remplacer le RAG mots-clés par Supabase pgvector
  • Ajouter un pipeline d'embedding automatique (n8n → Gemini embedding-001 → Supabase)
  • Chunking sémantique par section (procédure / symptôme / cause)
  • Garder toute l'architecture existante

Ce qu'on n'a toujours pas

  • Architecture multi-agents
  • Score de confiance
  • Feedback loop opérationnel
  • Évaluation de qualité
Quand choisir A

Si l'urgence est uniquement la scalabilité du RAG de Nicolas et que Vianney peut attendre. Bon choix si les deux projets restent séparés.

B
Convergence — Le meilleur des deux mondes ⭐ Recommandé
8–12 semaines · Risque modéré · Récompense élevée

Fusionner les deux projets en une infrastructure unifiée. Base opérationnelle de Nicolas + architecture agents de Vianney + vrai moteur vectoriel + ce qui manque aux deux.

Ce qu'on prend de Nicolas

  • DevOps complet (PROD/DEV + scripts + rollback)
  • Méthode de développement (METHODE_TRAVAIL_CLAUDE.md)
  • Pipeline ingestion Gemini Files API éprouvé
  • Error handlers sur tous les nœuds critiques
  • Mémoire conversationnelle Telegram
  • Documentation exhaustive

Ce qu'on prend de Vianney

  • Architecture multi-agents (Superviseur → sous-agents)
  • Pattern agentTool / toolWorkflow / $fromAI()
  • Score de confiance automatique par fiche
  • Organisation Drive auto-gérée par l'IA
  • Ingestion binaire directe (PDF, audio)

Ce qu'on ajoute (absent des deux)

  • Chunking sémantique par section
  • Supabase pgvector (remplace BigQuery et mots-clés)
  • Hybrid search BM25 + vectoriel + RRF fusion
  • Golden dataset 20–30 questions + évaluation automatique
  • Feedback 👍/👎 Telegram → réinjection ranking
  • Détection d'incertitude (seuil similarité)
⭐ Recommandé — Pourquoi

C'est le seul scénario qui capitalise sur les 15+ sprints de Nicolas sans les jeter, intègre l'architecture plus avancée de Vianney, et ajoute les fondations manquantes aux deux. Il prépare aussi la migration vers Gemini Embedding 2 dès sa sortie GA.

C
Greenfield avec microservice RAG dédié
16–20 semaines · Risque élevé · Récompense maximale

n8n reste uniquement pour les déclencheurs et les intégrations (Airtable, Telegram, Drive). Toute la logique RAG est extraite dans un microservice Python avec une API REST.

Avantages

  • Logique RAG testable unitairement (pytest)
  • Évaluation RAGAS automatique (faithfulness, relevancy, context precision)
  • Observabilité complète (Langfuse — chaque requête tracée)
  • Indépendant du cloud (Qdrant self-hosted, open source)
  • CI/CD IA avec tests de régression automatiques

Pourquoi pas maintenant

  • Requiert des compétences Python non acquises
  • Nouveau service à déployer et maintenir
  • 4+ mois sans amélioration de la prod
  • Risque de sur-ingénierie pour le volume actuel
Décision future

Ce scénario est la cible à 12–18 mois si le volume dépasse 500 fiches ou si la complexité des agents le justifie. L'architecture du Scénario B doit anticiper cette migration en isolant la logique RAG dans des outils n8n clairement délimités.

Section 9

Gemini Embedding 2 — Analyse de l'opportunité

⚡ Sorti le 10 mars 2026 — 3 jours avant cette analyse

Premier modèle d'embedding nativement multimodal. Texte, images, vidéos, audio et PDF dans un seul espace vectoriel unifié. C'est exactement le cas d'usage des deux projets.

Spécifications techniques

  • Model ID : gemini-embedding-2-preview
  • Dimensions : 128 à 3 072 (Matryoshka MRL)
  • Recommandé : 768, 1536 ou 3072 dims
  • Input texte : 8 192 tokens (vs 2 048 pour embedding-001)
  • Multilingue : 100+ langues
  • Task types : RETRIEVAL_DOCUMENT, RETRIEVAL_QUERY, SEMANTIC_SIMILARITY, CLUSTERING…

Limites médias (contraignantes)

  • Images : 6 par requête
  • Vidéo : 128 secondes max
  • Audio : 80 secondes max
  • PDF : 6 pages max
  • Statut : Preview — pas production-ready
  • Incompatible avec text-embedding-004 → re-embedding requis

Ce que ça change concrètement

Avant (pipeline actuel)

Vidéo → Gemini Flash (transcrit)
     → Texte brut
       → text-embedding-004
         → Vecteur texte
// 2 étapes, information perdue

Après (Phase 2)

Vidéo → gemini-embedding-2
     → Vecteur (même espace que texte)
// 1 étape, information visuelle préservée
→ Décision recommandée

Ne pas l'utiliser aujourd'hui. Les limites médias (128s vidéo, 6 pages PDF) sont trop contraignantes pour les cas d'usage réels. L'architecture à poser maintenant doit anticiper la migration : tables Supabase avec colonne embedding_model_version, pour basculer progressivement sans casser la prod dès que le modèle passe GA.

Matryoshka Representation Learning (MRL) — Pourquoi c'est important

Les modèles avec MRL permettent de stocker une fois à haute dimension (3072) et de chercher à basse dimension (768) pour la vitesse. Le score de similarité est ensuite raffiné avec les dimensions complètes pour les top-K résultats. C'est une optimisation coût/qualité native — Gemini embedding-001 et Embedding 2 le supportent tous les deux.

Section 12

Décisions à prendre lors du sprint commun

1

Valider le Scénario B comme direction commune

Les deux projets convergent vers une infrastructure unifiée, ou restent parallèles avec des échanges ponctuels ? Le Scénario B est plus ambitieux mais produit un système plus solide et maintenable à long terme.

2

Choisir Supabase comme vector DB commune

Remplacement de BigQuery (Vianney) et du RAG mots-clés (Nicolas) par Supabase pgvector. Instance unique, partagée, ou deux instances séparées par usage ? Supabase Free tier suffit pour <500MB de vecteurs (~200 000 chunks).

3

Définir le schéma de chunking commun

Les fiches des deux projets ont des structures différentes (torréfaction vs distributeurs). Un schéma commun de chunking par section est-il applicable aux deux, ou faut-il deux variantes ?

// Proposition de schéma universel
{
  chunk_id, fiche_id, projet,    // identifiants
  section,                       // "titre" | "procedure" | "symptomes" | "causes" | "materiel" | "notes"
  machine, type_intervention,    // métadonnées filtrables
  contenu,                       // texte du chunk
  vecteur[],                     // 768 ou 3072 dims
  embedding_model_version,       // anticipe migration Embedding 2
  score_confiance,               // 0-100 hérité de l'agent Superviseur
  created_at
}
4

Adopter la méthode de développement Nicolas pour les deux projets

METHODE_TRAVAIL_CLAUDE.md + CLAUDE.md + structure de documentation → à appliquer au projet Vianney dès le premier sprint commun. Sans ça, la dette technique de Vianney se reconstituera sprint après sprint.

5

Construire le golden dataset ensemble

20–30 questions réelles avec réponses de référence — Nicolas et Vianney contribuent chacun leurs cas d'usage. C'est 2 heures de travail qui rendront chaque sprint futur mesurable.

Synthèse en une phrase

Deux POCs qui ont chacun résolu la moitié du problème : Nicolas a la fiabilité, Vianney a l'architecture — le sprint commun consiste à les faire se compléter plutôt que coexister, en ajoutant les fondations RAG qui manquent aux deux.
